<p style="text-align: center;">RESTful&nbsp;설계방식</p>
<hr contenteditable="false" />
<p style="text-align: center;">RESTful 설계 방식은 웹 API를 자원중심으로 설계하고</p>
<p style="text-align: center;"><span style="letter-spacing: 0px;">HTTP의 표준 규칙을 최대한 활용하는 아키텍처 스타일입니다.</span></p>
<p style="text-align: center;">&nbsp;</p>
<p style="text-align: center;"><span style="letter-spacing: 0px;">단순히 REST API가 아니라 REST의 원칙을 잘 지킨 API를 보통 RESTful하다고 표현합니다.</span></p>
<hr contenteditable="false" />
<p style="text-align: left;">&nbsp;</p>
<h3 style="text-align: left;"><span style="letter-spacing: 0px;">REST(Representational State Transfer)의 핵심 원칙</span></h3>
<p><span style="letter-spacing: 0px;">다음과 같은 핵심원칙으로 RESTful 설계를 합니다.</span></p>
<p>&nbsp;</p>
<p><b> Client&ndash;Server 구조</b></p>
<p>클라이언트: UI, 사용자 요청</p>
<p>서버: 비즈니스 로직, 데이터 관리</p>
<p>서로 역할 분리 -&gt;확장성을 향상</p>
<p>&nbsp;</p>
<p><b> Uniform Interface </b></p>
<p>제일 중요한 원칙입니다.</p>
<p>자원은 URI로 식별하고 행위를 URI에 넣지 않아야합니다.</p>
<pre class="bash" id="code_1767524236292"><code>/users/1 O
/posts/10/comments O

/getUser X
/deletePost X</code></pre>
<p>&nbsp;</p>
<p><b> 자원 기반 설계(Resource-Based)</b></p>
<p>모든 것을 자원(Resource)으로 표현합니다.</p>
<p>예를 들어 사용자, 게시글, 댓글 등이 각각 하나의 자원이 되며 각 자원은 고유한 URI를 통해 식별됩니다.</p>
<p>/user/123처럼 명사 형태로 자원을 표현하는 것이 일반적입니다.</p>
<p>&nbsp;</p>
<p><b>HTTP 메서드 활용</b></p>
<p>자원에 대한 행위는 HTTP 메서드로 표현합니다.</p>
<p><figure class="imageblock alignLeft"><span><img height="326" src="https://blog.kakaocdn.net/dn/X7FIP/dJMcai9MAHf/oYZPC2U4agV3v8PT1mAkrk/img.png" width="521" /></span></figure>
</p>
<p>=&gt; 멱등성 : 같은 요청을 여러번 보내도 결과가 동</p>
<p>&nbsp;</p>
<p>이를 통해 URI는 자원만 나타내고, 행위는 메서드가 담당하게 됩니다.</p>
<p>&nbsp;</p>
<p><b> 무상태성(Stateless)</b></p>
<p>서버는 클라이언트의 상태를 저장하지 않습니다.</p>
<p>각 요청은 독립적이며 필요한 모든 정보를 포함해야 함으로써 서버의 확장성을 높이고 시스템을 단순하게 만듭니다.</p>
<p>&nbsp;</p>
<p><b>계층화된 시스템</b></p>
<p>클라이언트는 대상 서버에 직접 연결되었는지, 중간 서버를 통해 연결되었는지 알 수 없습니다.</p>
<p>이를 통해 로드 밸런서, 프록시, 게이트웨이 등을 자유롭게 배치할 수 있습니다.</p>
<hr contenteditable="false" />
<h3>RESTful한 URI 설계</h3>
<p>좋은 RESTful API는 직관적이고 일관된 URI 구조를 가집니다.</p>
<p>&nbsp;</p>
<p><b> 자원은 복수 명사로 표현 </b></p>
<p>/users는 사용자 목록을, /users/123은 특정 사용자를 의미합니다.</p>
<p>동사보다는 명사를 사용하며, /getUsers가 아닌 GET/users처럼 HTTP 메서드로 행위를 표현합니다.</p>
<p>&nbsp;</p>
<p><b> 계층 관계는 URI의 경로 </b></p>
<p>/users/123/posts는 특정 사용자의 게시글들을, /posts/456/comments는 특정 게시글의 댓글들을 의미합니다.</p>
<p>&nbsp;</p>
<p><b> 쿼리 파라미터 활용</b></p>
<p>필터링, 정렬, 페이징 같은 부가 기능은 쿼리 파라미터를 활용합니다.</p>
<p>/posts?category=tech&amp;sort=date&amp;page=2처럼 표현할 수 있습니다.</p>
<hr contenteditable="false" />
<h3>상태 코드와 응답 설계</h3>
<p>HTTP 상태 코드를 적절히 활용하는 것도 RESTful 설계의 중요한 부분입니다.</p>
<p><figure class="imageblock alignLeft"><span><img height="606" src="https://blog.kakaocdn.net/dn/ByOgV/dJMcagRHVfa/ts6svBA0meKMWdK3bMi6MK/img.png" width="651" /></span></figure>
</p>
<h3>&nbsp;</h3>
<hr contenteditable="false" />
<h3>애매한 경우</h3>
<p><b> 로그인 / 로그아웃</b></p>
<p>로그인은 행위처럼 보이지만 REST는 동사를 URI에 쓰면 안됩니다.</p>
<p>&nbsp;</p>
<p>따라서 로그인 방식인 세션과 토큰 방식에서 세션과 토큰을 자원으로 간주합니다.</p>
<pre class="bash" id="code_1767524500021"><code>!세션
POST   /sessions        // 로그인
DELETE /sessions/{id}   // 로그아웃

!토큰
POST   /auth/token // 로그인
DELETE /auth/token // 로그아웃</code></pre>
<p>-&gt; 행위처럼 보이면 그 결과로 생기는 상태나 리소스를 찾습니다.</p>
<p>&nbsp;</p>
<p>&nbsp;</p>
<p><b> 좋아요/찜/팔로우 같은 기능</b></p>
<p>버튼은 토글인데 REST는 명확한 상태 변화를 선호합니다.</p>
<p>&nbsp;</p>
<p>따라서 관계를 자원으로 모델링 합니다.</p>
<pre class="bash" id="code_1767524674932"><code>POST   /posts/1/likes      // 좋아요
DELETE /posts/1/likes     // 좋아요 취소</code></pre>
<p>-&gt; 토글기능 도 결국 생성과 삭제입니다.</p>
<hr contenteditable="false" />
<h3>RESTful 설계의 현실적인 원칙</h3>
<p>실무에서는 이렇게 정리합니다</p>
<p>&nbsp;</p>
<p><b>"완벽한 REST보다 의도가 명확한 API가 더 중요하다"&nbsp;</b></p>
<hr contenteditable="false" />
<h3>실무&nbsp;기준&nbsp;우선순위</h3>
<p>(1)&nbsp;URI&nbsp;의미가&nbsp;명확한가? <br />(2)&nbsp;HTTP&nbsp;Method&nbsp;의미에&nbsp;맞는가? <br />(3)&nbsp;프론트/백엔드가&nbsp;이해하기&nbsp;쉬운가? <br />(4)&nbsp;확장&nbsp;가능하고&nbsp;유지보수&nbsp;가능한가?</p>
<hr contenteditable="false" />
<h3>마무리</h3>
<p>완벽하게 REST 원칙을 모두 지키는 것보다는, 팀의 상황과 프로젝트의 특성에 맞게 유연하게 적용하는 것이 중요합니다.</p>
<p>핵심은 일관성 있고 예측 가능한 API를 만드는 것입니다. 명확한 문서화와 함께 제공하면, 다른 개발자들이 쉽게 이해하고</p>
<p>사용할 수 있는 API가 됩니다.</p>
